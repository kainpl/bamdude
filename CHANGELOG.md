## [Unreleased]

### Added

- **Connection Diagnostic for "printer won't connect / won't print".** A one-click checklist that runs the checks a maintainer would do by hand — TCP reachability of the MQTT (8883), FTPS (990) and camera (322) ports, Docker network mode, printer/host subnet match, MQTT credentials, and LAN Developer Mode — and explains each failure with a concrete fix. Run it from the printer card's action menu, a quick button that appears on an offline printer, or the new **Connection Diagnostic** section on the System page; the in-app bug reporter also scans your printers when the form opens and shows the result inline so a setup problem can be self-fixed before filing a report. en + uk. (Upstream Bambuddy #1465 triage / 76e327f4.)
- **Event-loop stall watchdog.** A frozen asyncio loop used to be invisible — no log line, `/health` hangs, the process may ignore SIGTERM. A heartbeat now re-arms a `faulthandler` timer that dumps every thread's stack to stderr if the loop stalls for 30s, so a hard freeze shows up in `docker compose logs` with the blocked frame instead of nothing. (Upstream Bambuddy #1486 / 774eba73.)

- **Cross-nozzle-class re-slicing (single ↔ dual nozzle).** Re-slicing a 3MF built for a single-nozzle printer onto an H2D/H2C/X2D (or vice-versa) now works: BamDude detects the class crossing and forwards the slicer's `--arrange` so BambuStudio repositions objects for the target bed and reconciles the embedded project settings, instead of failing on the source's coordinate layout. Multi-plate "slice all" across classes slices each plate independently and merges them into one 3MF. Per-slot filament picks for slots a plate doesn't use are normalised so the loaded-filament temperature validator doesn't reject the slice. (Upstream Bambuddy #1493.)
- **Re-slice a saved 3MF for a different printer.** The Slice dialog no longer blocks (or warns) when the 3MF was originally built for another printer model — the slicer re-slices it for whatever printer the chosen profile/bundle targets (bed, kinematics, nozzle count and start-gcode all come from the target). Removed the obsolete "printer mismatch" gate and its now-dead `source_printer_model` plumbing. (Upstream Bambuddy #1325 / ed27b27a.)
- **The Slice dialog filters process & filament profiles to the selected printer.** Picking a printer now narrows the process and filament dropdowns to the profiles compatible with it (by the profile's own `compatible_printers`, then by any uploaded Slicer Bundle, then by the `@BBL <model> <nozzle>` naming convention — nozzle-aware, so a 0.6-nozzle profile won't show for a 0.4 printer). Profiles for other printers drop into a trailing **"Other printers"** group rather than disappearing, and profiles with no printer info are never hidden. The printer and process dropdowns also **default to the profiles the 3MF was prepared with**, and switching printer re-picks any selection left incompatible. The same matcher now drives the **filament-calibration** wizard's profile picker (model **and** nozzle aware, where it was model-only before), there hiding incompatible profiles outright since a wrong-printer calibration print wastes a bed. (Upstream Bambuddy #1325 + follow-ups.) en + uk.
- **Belize Dollars (BZD) added to the cost-currency dropdown** (Settings → Cost). `getCurrencySymbol('BZD')` → `BZ$`. (Upstream Bambuddy #1454.)
- **K-profiles auto-link to spools by filament.** Once a printer is calibrated for a filament, every spool of that filament now picks up the K-profile automatically — no per-spool clicking. Matching is by the spool's slicer preset, resolved to its **base** for custom filaments (e.g. a custom "Generic PETG"-based filament inherits the Generic PETG calibration), across all your printers and nozzles. The chosen profile is auto-activated when none is active yet, manual PA-tab links always win, and each spool card now shows its linked K-profiles with an "auto" marker. Custom filaments also now correctly surface matching K-profiles and colours in the AMS slot dialog and the spool's PA-Profile tab (the base preset is resolved via the cloud `base_id`). en + uk.
- **"Clear SD card" in the printer file browser.** A one-click recursive wipe that deletes every file on the SD card and removes all nested subfolders, while keeping the standard top-level folders (cache, model, timelapse…) the printer expects to exist. Guarded by a danger confirmation, and refused while a print is running or paused (the active job streams from the card). The whole walk runs over a single FTP connection so it stays fast even on a card full of cached models. Requires the `printers:files` permission. en + uk.
- **Click a file name to see its print history.** In the File Manager (grid and list views) and on the MakerWorld → History tab, the file name is now a link that opens the Archive filtered to every print dispatched from that library file, with a dismissible chip showing which file you're drilled into. Backed by a new `library_file_id` filter on the archive list (the link is stamped at dispatch time, so external/manual prints never match). en + uk.
- **Anonymized usage telemetry (opt-out).** BamDude now sends a once-a-day anonymized snapshot — app version, OS platform/arch, Docker yes/no, aggregate counts (archives, printers, spools, projects, smart plugs, **users**, **AMS units**), printer **models** (never names/serials), enabled integrations (Spoolman, Obico, Telegram, OIDC, Git backup, slicer API) and daily print counts — to help prioritise development. It is keyed by a random `install_id` generated once at first boot (`DATA_DIR/.install_id`, mode 0600) and **not** linked to any account, email, IP or hardware id. The id is included in **backups** (so a restore keeps the same identity) and in the **support bundle / bug report** (so a report can be correlated with telemetry). **No IP is stored** by the collector (a coarse country is derived from the proxy header, then discarded), and no names, serials, file paths or settings values are ever sent. It is **on by default with disclosure** — a notice on the first-run setup screen, and a toggle under **Settings → General** (en + uk). Turning it off immediately asks the collector to erase this install's data. Disable entirely with `TELEMETRY_DISABLED=true`, or point it at your own collector with `TELEMETRY_RELAY_URL`. The daily send is fail-silent and never disrupts the app. Backend collector + admin dashboard live in the separate `bamdude-telemetry` project.

### Fixed

- **A wrong-cased printer serial no longer connects to silence.** Bambu serials are uppercase and the MQTT report topic `device/<serial>/report` is case-sensitive, so a serial typed in the wrong case (or with stray whitespace) connected and subscribed fine but never received a single status report — every field stayed "unknown". The serial is now normalised (trimmed + uppercased) on save, and a connection that keeps going stale without ever receiving a report logs a one-shot hint pointing at the serial number instead of looping silently. (Upstream Bambuddy #1465 / 01787eb1.)
- **BamDude behind Cloudflare no longer needs `unsafe-inline` to load.** Cloudflare injects a bot-detection script whose hash rotates per request, so it could never be allowlisted under the strict `script-src 'self'` CSP. The SPA policy now also stamps a fresh per-request `nonce` — Cloudflare clones that nonce onto its injected script so it passes, without weakening the policy (`'self'` stays, no `'unsafe-inline'`). Also `/manifest.json`, `/sw.js` and `/sw-register.js` now answer `HEAD` (uptime probes / `curl -I` were getting 405). (Upstream Bambuddy #1460 follow-up.)
- **Re-slicing an archive for a different printer no longer mislabels it with the source printer.** A cross-model re-slice inherited the source archive's `printer_id`, so the new archive card and reprint modal showed the original printer's name; the id is now cleared on a cross-model re-slice (the card falls back to the sliced-for model). (Upstream Bambuddy c0c07bc5.)
- **"Prefer Lowest Remaining Filament" now uses your inventory weight, not just the printer's RFID counter.** The dispatch sort read only the MQTT `remain` value, which is meaningful only for Bambu RFID spools (non-RFID spools report `-1`, so multiple trays tied and the lowest AMS slot always won) and is the printer's counter rather than BamDude's `label_weight − weight_used` (or Spoolman's `remaining_weight`). When a slot is bound to an inventory spool, that tracked remaining is now the sort signal — inventory-tracked spools are preferred over untracked ones, then by ascending remaining. Works in both internal-inventory and Spoolman modes; un-bound slots keep the old MQTT behaviour. (Upstream Bambuddy #1508.)
- **The 3D preview no longer freezes the page on complex multi-part 3MFs.** Parsing a large, many-component model (e.g. a multi-colour "parted" MakerWorld statue) ran as one long synchronous burst that locked the UI — including the preview's own close button — until it finished. The parser now yields to the browser periodically so the modal stays responsive. (Upstream Bambuddy #1412.)
- **The library 3D preview's slice button now respects the Slicer API setting.** With in-app slicing enabled, the preview opened the *external* slicer while the file-row action used the in-app Slice dialog; the preview now offers the same in-app "Slice" action for sliceable files. (Upstream Bambuddy #1325 follow-up.)
- **Library files now show their filename, not the embedded 3MF title.** A file imported from Bambu Studio or MakerWorld would display its in-app project title (often the generic "Exported 3D Model") on the card, in search and in sort, instead of the filename you saved it as — fixable only by a rename round-trip. All import paths (upload, ZIP, external-folder scan, MakerWorld, slice output) now drop the embedded title on the way in, and a one-time migration clears it from files imported before the fix. Print archives still keep their real print name. (Upstream Bambuddy #1489.)
- **The File Manager "All Files" view now shows files inside subfolders.** It was requesting only root-level files, so a library where every file lived in a subfolder appeared empty under "All Files". (Upstream Bambuddy #1499.)
- **AMS drying no longer powers the printer off seconds after it starts.** A tray-only AMS status update (which carries no drying fields) was dropping the remaining `dry_time` during the state merge; the drying-complete detector then read the absent value as 0, saw a false "60 → 0" finish, and — for anyone using the new smart-plug auto-off-after-drying — cut power mid-cycle. The merge now preserves `dry_time`/humidity/temp across tray-only updates, and the completion detector only fires on an explicit `dry_time` reading. (Upstream Bambuddy #1462.)
- **A malformed MQTT PUBLISH from a Bambu printer no longer kills the connection.** The printer's broker occasionally delivers a truncated packet whose declared topic length exceeds the bytes present; paho-mqtt would build a negative `struct` count and crash its network thread with `struct.error: bad char in struct format`, dropping the printer offline until the ~70 s stale-connection watchdog reconnected. BamDude now drops the malformed push (the printer re-pushes its state within seconds), mirroring the guard already in place for malformed SUBACK packets.

### Changed

- **A failed slice now shows a dismissible dialog with the slicer's reason** instead of a toast that auto-vanished before you could read it. (Upstream Bambuddy #1493.)
- **File library multi-select now toggles only when you click the checkbox**, not anywhere on the card or row. Clicking a card/row body no longer flips its selection (which made it easy to select files by accident while scanning the library); the checkbox is the single select affordance, matching the printers page. Applies to both grid and list views.

### Security

- **Pinned `starlette>=1.0.1` and `idna>=3.15`** against PYSEC-2026-161 (Starlette) and CVE-2026-45409 (idna ReDoS) — both are transitive dependencies that were resolving to the vulnerable versions. (Upstream Bambuddy security pins.)

## [0.4.5b3] - 2026-05-16

Third beta of the 0.4.5 cycle. The Filament Calibration wizard's Wave 1/2 lands — PA Tower, PA Pattern and PA Line now run end-to-end as real calibration prints (Python ports of the BambuStudio generators), with per-mode preset overrides applied through the slicer sidecar. Plus a K-profile data-layer pass (external-spool pre-print bind, in-place sync, BS-parity edit/delete, hard-prune of printer-side deletes), per-printer queue pause that works mid-print, an auto-queue lifecycle + stats cleanup, and honest client identification on every Bambu Lab / MakerWorld call. Image: `ghcr.io/kainpl/bamdude:0.4.5b3` / `kainpl/bamdude:0.4.5b3`. Pin the exact tag — `:latest` still tracks 0.4.4.1.

### Added

- **Add-to-queue dialog hides printers whose queue is paused.** Paired with the queue pause feature — a paused queue refuses new items, so the "Add to queue" modal's printer picker now drops any printer whose queue is paused (the backend would `409` the add anyway). Only the add-to-queue mode filters; reprint / edit-queue-item are untouched.

- **Per-printer queue pause/resume — works even while a print is running (m067).** A printer's queue could only be paused when idle: the pause/resume control was hidden during a print, and the API rejected pausing a `printing` queue outright. The block was structural — `PrinterQueue.status` is a single field the scheduler reads as the authoritative "printer busy" marker, so it couldn't also hold a pause state. New `PrinterQueue.is_paused` flag is orthogonal to `status`: a queue can be `printing` and `is_paused` at once. Pausing a queue now (a) leaves any running print completely alone — it finishes normally — (b) stops the next item from dispatching, (c) refuses new items being added (manual add returns `409`, and the auto-queue scheduler skips the printer entirely so it can't route work there either). The queue card's pause/resume button is now shown in every state with a "Paused" pill when active; resume also clears a status-level `paused`/`error` state in the same click. Migration m067 adds the column.

- **Auto-queue panel shows archive-backed totals.** The auto-queue panel listed only pending router items — there was no at-a-glance "how have auto-queue prints been doing" the way every per-printer queue card has a `done · failed · cancelled` footer. New `GET /auto-queue/stats` counts `print_archives` flagged `from_auto_queue` and folds `aborted` / `stopped` into `cancelled`, mirroring `get_queue_terminal_counts` exactly. The panel renders a footer line (`done · failed · cancelled`) once at least one auto-queue print has finished. Because the `auto_queue_items` row is deleted when its print finishes, this count comes entirely from the archive flag — the panel and the per-printer queue cards now read their history from the same source.

- **Auto-queue print provenance on archives (`print_archives.from_auto_queue`, m066).** An `auto_queue_items` row is a pre-dispatch *router* — once the `AutoQueueScheduler` copies it into a printer's `print_queue` and that print runs, the auto row carries nothing more. It used to linger forever as a `status='assigned'` row with an `assigned_to_item_id` that dangled the moment the completed `print_queue` item was auto-cleaned (`foreign_key_check` surfaced 7 such orphans). New lifecycle: when a `print_queue` item is deleted, its source `auto_queue_items` row is deleted with it (`detach_print_queue_refs` helper, wired into all three `print_queue` deletion sites — `on_print_complete` auto-cleanup, the manual `DELETE /queue/{id}` route, and per-user bulk delete). The lasting record moves to the archive: every print the scheduler dispatched is now stamped `from_auto_queue=True` at archive-creation time (from the queue item's `source_auto_item_id`). This lets the auto-queue view compute archive-backed completed / failed totals the same way per-printer queues already do via `get_queue_terminal_counts` — a simple stats endpoint + UI can land on top without further schema work. `calibration_session` rows referencing a deleted `print_queue` item get their `print_queue_item_id` nulled instead of deleted — a calibration's session history outlives its queue item. All cleanup is done with explicit `DELETE` / `UPDATE` rather than `ON DELETE` clauses because SQLite runs with `foreign_keys=OFF`. Migration m066 adds the column, backfills `from_auto_queue` for archives produced by an auto-queue print that's still traceable through a surviving non-pending `print_queue` row carrying `source_auto_item_id` (pending rows are skipped — `print_queue.archive_id` only points at the produced archive from dispatch onward; completed auto-prints lost their `print_queue` row on completion and are unrecoverable, so the backfill is best-effort and the flag is authoritative only for prints dispatched under the new code), and deletes the orphaned `auto_queue_items` backlog — rows whose `assigned_to_item_id` dangles because the `print_queue` row is gone. Auto rows whose `print_queue` item still exists are left alone: the operator deleting that queue item triggers the same `detach_print_queue_refs` cleanup, so the one-time migration never removes a row the user can still act on.

- **K-profile name is editable on the Profiles page (Bambu Studio parity).** The edit modal used to lock the profile-name field — operators could fix the K value but not relabel a slot. Field is now editable both on add and on edit. The DB row keeps its identity through the rename because the K-profile cache sync (see Fixed) now matches by combo `(printer, filament_id, nozzle, vol_type, extruder)` and mirrors `name` + `pa_k_value` onto the matched row — so attached notes (m065) stay linked across renames.

- **K-profile delete sends one BS-parity `extrusion_cali_del` shape for every printer.** `delete_kprofile` branched on a serial-prefix dual-nozzle guess: H2D / X2D / H2C got `extruder_id · nozzle_id · filament_id · cali_idx · nozzle_diameter`, while non-dual (X1 / P1 / A1 / P2S) got the same plus a `setting_id` field. BambuStudio's `command_delete_pa_calibration` (DeviceManager.cpp:1905) sends one fixed payload for all models — the dual-nozzle field set, no `setting_id`. BamDude now matches: single shape, no branch, no `setting_id`. (BS additionally appends `nozzle_pos` / `nozzle_sn` when an H2D nozzle rack is mounted — BamDude doesn't wire the nozzle-rack feature anywhere yet, so those stay omitted.)

- **K-profile edit is now a single in-place `extrusion_cali_set`, not delete + add (BS parity).** The non-H2D edit path used to publish `extrusion_cali_del` → `await asyncio.sleep(0.5)` → `extrusion_cali_set` with a fresh `setting_id` — which left a noticeable window where the slot didn't exist on the printer and a fresh `setting_id` broke any reference (e.g. tracked auto-cali history). - **Calibration cache per-printer-instance, not per-model (m063).** `filament_calibration` rows keyed by `printer_id`. Two X1Cs in a farm carry independent K values for the same material; the History modal scopes to that specific printer.

- **Cache mirrors the printer's live K-profile list, event-driven.** Sync runs on every MQTT (re)connect and whenever the printer's list actually changes (hash-diff filtered so it doesn't hammer the DB on every `push_status`). New rows arrive `is_active=false` — you still explicitly promote one row per `(material, nozzle, extruder)` combo. UI fetches stay consistent; you don't have to open the manage dialog for BamDude to catch up after reconnect / on-printer save.

- **Cache row carries printer-side `nozzle_id`.** Every row stores the raw 4-char identifier (`HS00-0.4`, `HH00-0.6`, …). P1S / A1 / A1 mini don't ship a per-profile value — BamDude derives it from the live nozzle hardware state so the column is filled regardless of model.

- **K-profile notes survive printer restarts (m065).** Notes re-keyed to BamDude's stable `filament_calibration.id`. The old `setting_id` keying drifted across restarts and silently re-attached notes to the wrong profile. Existing notes are cleared as part of m065 — clean slate beats fuzzy recovery on data that was already unreliable.

- **Spool ↔ K-profile thin link rows (m064).** `spool_k_profile` + `spoolman_k_profile` drop their copy of `k_value` / `name` / `cali_idx` / `setting_id` / `nozzle_type` / `nozzle_diameter` in favour of a single `filament_calibration_id` FK. One hundred PETG spools sharing the same calibration → one cache row + 100 links instead of 100 duplicated K rows. PA tab UI unchanged.

- **`calibrated_on_printer_id` dropped.** After m063 keyed `filament_calibration` on `printer_id` the column was always equal to it — pure duplicated state. Audit code now reads `printer_id` directly; printer-sync rows stamp `calibrated_by_user_id` with the first admin user id as the "came from printer, not wizard" marker.

- **Archive filter is now `All / Test / Regular` keyed on `is_calibration`.** The old `All files / Sliced GCODE / Source only` filter was meaningful when archive held arbitrary uploads — but archive has been print-history-only since 0.4.2. Query param renamed `file_type` → `kind ∈ {all, calibration, regular}`; localStorage key migrated.

### Fixed

- **"Printer Parts" tab shows nozzle hardware on P1S / A1 / A1 mini.** Those models push the nozzle as a canonical long name (`stainless_steel`, `hardened_steel`, …); our parser only understood the X1-family 4-char code (`HS00`, …) and rendered the long string raw with an empty Flow row. Parser is now a one-to-one port of BS `DevNozzleSystem::s_parse_nozzle_type` — long names (six categories: stainless / hardened / tungsten carbide / brass / E3D / undefined), 4-char codes, and the literal `"N/A"` sentinel. Flow class (`standard` / `high_flow` / `tpu_high_flow`) stored separately on `NozzleInfo` and surfaced on `/printers/{id}/status`.

- **Configure-AMS-slot dialog matches Generic PETG (and other Generic profiles).** Match was by name substring — works for "PLA Basic" but breaks for "Generic PETG" because the cloud-preset name lacks the literal "GENERIC". Match is now by `filament_id` equality with `GFS*` cloud ↔ `GF*` slot normalisation; name substring kept as a legacy fallback.

- **K-profiles edit dialog showed wrong flow type.** Three off-by-one defaults (`'HH00'` instead of `'HS00'` + an inverted ternary) made every existing row open as "High Flow". Existing rows now open with the flow type they were saved with.

- **Single-external slot binds K-profiles correctly.** External tray ID was `254`; BS standard is `255` (`VIRTUAL_TRAY_MAIN_ID`). With `254` the printer silently dropped every `extrusion_cali_sel` aimed at the single-external slot.

- **K-profile row listing on dual-extruder machines.** Handler read non-existent `kp.extruder_id` (correct field is `kp.extruder`) — extruder column was empty for every row.

- **Manual PA wizard captures the printer's assigned `cali_idx` on save.** Wizard now round-trips `extrusion_cali_get` after save_result and stashes the live slot index, so the very next print binds without waiting for an unprompted firmware push.

- **Sync mirrors `filament_setting_id` from the printer.** The printer regenerates `setting_id` on every `extrusion_cali_set`; sync used to update only `cali_idx` and `nozzle_id`, leaving DB drift untouched even when the live value differed. Now mirrored whenever the printer reports a non-empty value that differs from cache.

- **Edit K-profile "Save" with no printer-relevant change no longer hits the printer.** Saving only a note — or clicking Save without changing anything — used to fire `extrusion_cali_set` and force the printer to regenerate `setting_id`. The form now diffs `name` / `k_value` / `filament_id` / `nozzle_id` / `nozzle_diameter` against the loaded profile; identical → note saved locally, no MQTT publish.

- **`/printers/{id}/kprofiles/` writes audited.** Every add / edit / batch-add / delete writes a `calibration_audit` row (`action ∈ kprofile_{add, edit, batch_add, delete}`) with full payload + user id + success / error. Closes the trace gap where mutations from the K-profiles UI page were invisible to forensic queries.

- **AMS humidity / temperature history modal** — missing uk i18n keys + dialog now follows the active theme background variant (warm / cool / oled / slate / forest).

- **On-disk archive copies are now always the unpatched original; patched bytes never land in `archive/`.** Pre-fix `archive_print` wrote `dispatched_file` (post-patch) to disk so its hash matched `content_hash`, but that made reprint-with-toggled-patches a silent no-op: `gcode_patcher`'s `_M970_RE` only matches uncommented lines and can't undo a previously-baked comment, so re-running the patcher against an already-patched on-disk source returned passthrough and shipped the old patched bytes again. Two-hash split now decouples the roles: `content_hash` = SHA256(`dispatched_file`) — drives `on_print_start` restart-recovery (we hash the printer's copy and look for `content_hash == temp_hash`); `source_content_hash` (and the on-disk file at `file_path`) = SHA256(`source_file`) — the unpatched original reprint can re-feed to the patcher in either direction. `attach_3mf_to_archive` (FTP-recovery for fallback rows) now joins the same model — when the chain root already has an on-disk unpatched copy from a prior dispatch, it reuses that `file_path` instead of writing the printer's patched copy. Cross-printer file dedup moved from strict `content_hash` match to `effective_hash = COALESCE(source_content_hash, content_hash)`: every row sharing an unpatched origin shares one on-disk file, regardless of which patches each individual dispatch applied. Patched temp files continue to live only in `/tmp/bamdude_patch_*` for the FTP-upload window and are cleaned up by `background_dispatch.py:941` (archive flow) / `:1411` (library flow). Tests in `test_archive_dispatched_file.py` updated to assert the new on-disk-equals-source invariant.

- **New `scripts/prune_orphan_archive_files.py` reconciles `archive/` against the DB.** Walks `<DATA_DIR>/archive/` and compares every regular file's relative path to the union of `print_archives.{file_path, thumbnail_path}` + `library_files.{file_path, thumbnail_path}` (including trashed rows so the retention sweeper still has its inputs). Files not referenced are orphans (e.g. patched copies left behind by the previous semantics); empty directories are collapsed bottom-up. Default mode is dry-run; pass `--apply` to actually delete. Skips `archive/temp/` (runtime FTP staging) and stays inside `archive/` (other directories — `certs/`, `projects/`, `virtual_printer/` — are out of scope). SQLite-only — Postgres-backed installs are out of scope (the docstring suggests building the reference set directly via SQL).

- **`on_print_start` now closes stale `printing` archive rows up-front and skips the FTP+hash dance when the slicer says the print could plausibly still be running.** Previously a row could sit forever in `status='printing'` after a crash mid-print or a missed `on_print_complete`. The downstream name-match adoption block then revived such ghosts unconditionally — the next print of the same file (even with completely different bytes) would silently land back on the old archive instead of getting a fresh row. New helper `_close_stale_printing_rows(printer_id, check_name, db, logger)` runs at the top of `on_print_start` (after the `_active_prints` duplicate-guard, before `_expected_prints` lookup): it closes every stale row on this printer that's clearly not the live print — different name + `predicted_end < now` → `'completed'` with `completed_at = started_at + print_time_seconds` (slicer's predicted natural end), different name + `predicted_end > now` → `'cancelled'` (printer moved on). Same-name siblings older than the newest get the same treatment. The newest same-name row is preserved for the existing name-match adoption block — which is also tightened: only adopt if `started_at + print_time_seconds > now`. Net effect: when mid-print recovery hits a row whose slicer-predicted end is still in the future, the existing row gets adopted directly **without** the FTP-download + hash-compute step (saves tens of seconds on multi-MB 3MFs); when predicted end has already passed, fall through to the FTP/hash path so a slicer-underestimate live print is verified by hash and a "different bytes, same filename" reprint correctly creates a fresh row. Each closed row gets `extra_data['recovered_by_cleanup']=True` for audit. Name matching uses the lenient `_archive_matches_check_name` helper — `print_name == check_name` OR `filename` ∈ `{check_name, check_name+".3mf", check_name+".gcode.3mf"}` (with basename tolerance). This keeps the cleanup pass and the downstream name-match adoption block aligned, so legacy rows stored as `Plate_1.gcode.3mf` aren't treated as a "different file" when the new MQTT event reports `Plate_1.3mf` (or vice-versa). Trade-off: a long BamDude downtime followed by reprinting the same file from the printer's screen could see the new live print glued onto the old (still-`'printing'`) row downstream; documented as acceptable since the same risk exists in pre-cleanup-pass code via the unconditional name-match adoption that this feeds. Diagnostic INFO logs (`[cleanup] inputs:`, `[cleanup] row #...`, `[name-match] cand #...`) trace which rows were considered, why they were classified same/other-name, and the `predicted_end` vs `now` decision — left enabled at INFO since the volume is one burst per `print_start` event. 10 new integration tests in `test_close_stale_printing_rows.py` cover other-name completed/cancelled, same-name newest preservation, older-sibling closure, NULL `started_at` / `print_time_seconds` preservation, scoping by `printer_id`, terminal-status row skipping, empty-table no-op, and `.gcode.3mf` ↔ `.3mf` filename-variation regression.

### Fixed

- **File Manager card-mode three-dots menu no longer clips against the card edge on narrow viewports.** The card root carries `overflow-hidden` for the thumbnail crop, so the previously absolute-positioned dropdown got cut off at the card's right/bottom border whenever the file landed in the rightmost grid column on a small screen. Menu now portals to `document.body` (same pattern already in use for the list-mode `FileListActions` menu), with viewport-relative coords recomputed on scroll/resize. Default anchor is `bottom: viewportH − triggerTop + 4 px` so the menu's bottom edge sits exactly 4 px above the three-dots button regardless of how many items are visible — previously a fixed `estimatedHeight` left a noticeable gap when fewer items rendered. Flips to `top: triggerBottom + 4` when the trigger is too close to the viewport top to fit the menu above it. Width is now `w-max` (sized to the longest visible item via `whitespace-nowrap`) capped at `calc(100vw − 16 px)` instead of a hard-coded 240 px — the menu shrinks for files with few applicable actions and never overflows the viewport.

- **Auto-queue no longer double-stacks pending items onto the highest-id printer in a multi-printer fleet.** Reproducer: 4 idle printers eligible for the same auto-queue model, drop in 7 copies. Items 1-4 spread one-per-printer (P1..P4) on tick 1 — correct. Tick 2 was firing inside the small window between PrintScheduler's prep flipping `PrinterQueue.status='printing'` for P1/P2/P3 and the same flip landing on P4 (per-printer prep walks `(queue_id, position)` in order, and `ensure_fresh_connection_for_printer` can spend a few hundred ms per printer on stale MQTT sessions). The auto-queue's busy filter only checked `PrinterQueue.status='printing'`, so P4 still looked "free" and the next iteration of `find_eligible_printer` (which returns the first printer NOT in the busy set, by `printer.id` ASC) returned P4 unconditionally. Items 5/6/7 then queued behind item 4 on P4 — the other three printers, which had already started their prints, never saw new work even after their queues drained. Failure recovery on P4 made it worse: the stacked item started immediately and the next auto-tick repeated the same misrouting. Fix widens the busy seed in `auto_queue_scheduler.tick` to "PrinterQueue.status='printing' OR PrinterQueue has any pending PrintQueueItem"; routing now caps each tick at one new placement per printer and the next placement waits for the queue to genuinely drain. Side benefit: manual queue items, scheduled-future items, and items waiting for `manual_start` all count as "occupying" too — auto-queue stops piling on top of operator-queued work.

- **Queue dispatch now runs printers in parallel instead of one-after-the-other, while still honouring Staggered Start when it's enabled.** Adding the same file to six per-printer queues used to start one printer, wait for its full FTP / 3MF-patch / `register_expected` / MQTT-`project_file` round-trip, then start the next — a 4–6× wall-clock penalty on the first dispatch wave compared to a direct print of the same six files (which already went through `BackgroundDispatch._dispatcher_loop` → `asyncio.create_task` per job in parallel). Root cause: `print_scheduler.check_queue` did `await self._start_print(db, item)` in a serial `for item in items:` loop, and `_start_print` itself synchronously awaited `background_dispatch.run_from_queue_item` end-to-end. Refactored `_start_print` into a thin synchronous prep stage (file existence + status flip to `'printing'` + queue-counter update + AMS/options shape + stagger-slot pre-register) followed by a spawned `_dispatch_and_finalize` asyncio task that opens its own DB session and runs the full FTP/dispatch/watchdog/notification pipeline. The check_queue loop now returns from `_start_print` in milliseconds, so the next iteration can dispatch a different printer immediately. Stagger semantics preserved end-to-end via a new `print_scheduler.acquire_stagger_slot(printer_id)` helper invoked at the top of `background_dispatch._process_job` — idempotent for the queue path (the slot was pre-registered synchronously under `check_queue`'s serial loop, so the helper sees the printer already holds a slot and returns immediately) and gating for the direct-print path (cold acquire: polls every 2 s under an `asyncio.Lock` until a free slot is available, then registers — closes the prior gap where direct prints bypassed stagger entirely). Net effect: stagger off → six queue items now dispatch as fast as the dispatcher can start FTP sessions; stagger on with `concurrent=2` → both queue and direct dispatch waves admit two prints at a time, then wait for the configured interval (or bed-temp signal) before admitting the next pair.

- **"Printing" badge in the archives list no longer dead-ends on a black screen.** Both archive list views (gallery card + slim list row) wrapped the in-flight badge with `<Link to={`/printers/${archive.printer_id}`}>` — but the router has no `/printers/:id` route. The matching React Router didn't fall through to a 404, just rendered an empty Layout outlet. Every other "go to printer X" link in the codebase (queue card, project-detail timeline) uses `/#printer-${id}` — the printers list is the index route, the page already has a `useEffect` that scrolls to `id="printer-<id>"` once printers load. Switched both archive callsites to the same convention; updated the stale comment in `PrintersPage` that claimed the URL was `/printers#printer-<id>` (it's actually `/#printer-<id>`, which is what every existing caller uses).

- **`POST /auto-queue/` no longer returns 500 (and silently duplicates rows) when adding archive- or library-file-backed items.** The auto-queue response builder (`api/routes/auto_queue.py::_to_response`, originally landed in commit 81ae73f) read `item.archive.original_filename` and `item.library_file.original_filename` — neither model has that column. Both expose only `filename`. Every successful POST therefore committed the row, then raised `AttributeError` while serialising the response: client got 500, but the auto-queue table already had the new item. Operators retried (the toast read "Internal Server Error") and each retry duplicated the items — the dispatch flow then quietly worked through them, producing the "items keep showing up but the action button errored" symptom that surfaced from a 15-plate project dispatch (support bundle 2026-05-04 14:22). Fix mirrors the analogous builder in `print_queue._to_response`: archives surface `print_name or filename` (so multi-plate suffix is preserved), library files prefer the parsed `file_metadata["print_name"]` and fall back to the bare `filename`. Three new integration tests in `test_auto_queue_api.py` round-trip POST → GET against rows with archive / library_file relationships loaded — a future regression that touches `_to_response` will fail at 500 instead of slipping through into a release.

- **Restart-recovery adoption in `on_print_start` now disambiguates plates of multi-plate containers.** When a backend restart hit during a print of plate N from a multi-plate 3MF, both adoption paths (name-match block and post-FTP `content_hash` match) relied solely on `printer_id + status='printing'` to pick the right row — which assumes the cleanup pass left exactly one in-flight row per printer. `temp_hash` is identical across plates of the same container (the whole 3MF lands on disk + on SD regardless of which plate was selected), so a stuck `'printing'` row from a previously-crashed plate of the same container could be adopted in place of the live one — gluing the new run's progress / completion / energy tracking onto the wrong archive (and onto the wrong `plate_index`). Fix extracts a `live_plate_id` from the MQTT push (`parse_plate_id` over `data["filename"]` with `data["raw_data"]["gcode_file"]` fallback) at the top of `on_print_start`, then narrows both adoption queries with `(plate_index == live_plate_id OR plate_index IS NULL)`. The `IS NULL` branch keeps legacy archives (pre-m038) adoptable so installs upgrading from older builds aren't excluded. Cleanup helper `_close_stale_printing_rows` is intentionally **not** modified — it still closes by name only, so its rule "keep the newest same-name row" stays correct (the newest is virtually always the active plate); the plate-index check is layered on top in adoption, where it's load-bearing. Single-plate containers and `subtask_id` matches are unaffected (subtask_id is unique per dispatch). New diagnostic INFO log `[adopt] check_name=... live_plate_id=... (filename=... gcode_file=...)` traces the resolved plate index per event so a "should have been plate X but got Y" report can be triaged without a debugger.

- **Bulk dispatch no longer fails 4+ printers with `"Can't re-connect printer MQTT"` on a stale-connection race.** `printer_manager.connect_printer` used to wait a fixed `asyncio.sleep(1)` after `client.connect()` and then read `client.state.connected`. paho's `_on_connect` callback flips that flag asynchronously after the TLS handshake completes; on a single reconnect 1 second was almost always enough, but starting 4+ parallel TLS handshakes (the dispatcher hits `ensure_fresh_connection_for_printer` once per printer in the batch) plus concurrent 3MF patching pushed some handshakes past the 1s window — `state.connected` was still `False` when we returned, the dispatcher saw `False` from `ensure_fresh_connection_for_printer`, and the print was aborted with `RuntimeError("Can't re-connect printer MQTT")` from `background_dispatch.py:715` (archive flow) and `:1174` (library flow). Replaced the fixed sleep with an active poll: up to 10 s, 100 ms between checks, returns the moment `state.connected` flips. Healthy network: returns within the same 100-300 ms it always did. Genuinely-offline printer: still returns False, just after 10 s instead of 1 — slower failure is the trade-off for not spuriously aborting bulk prints when several handshakes happen to land on the same busy moment. Also added a WARNING log in `ensure_fresh_connection_for_printer` (printer name + id + IP) when the reconnect actually fails — earlier the operator just saw the generic "Can't re-connect" without knowing which machine timed out, since most callers re-raise without the printer context.

- **AMS slot hover-cards no longer slide under the navigation sidebar.** On the leftmost printer's leftmost slot, the centred `-translate-x-1/2` hover-card pushed past the left edge of `<main>` and got clipped by `overflow-auto`, visually overlapping the fixed sidebar. Both `FilamentHoverCard` (loaded slot) and `EmptySlotHoverCard` now measure the rendered card vs. the `<main>` bounding rect on show and apply a `marginLeft` shift that clamps the card inside the visible content area; the arrow pointer compensates with the inverse offset so it stays centred over the trigger. `marginLeft`-based shift was chosen over `transform: translateX` so the existing `animate-in zoom-in-95` enter animation keeps working.

Those callers are gone; any remaining caller (e.g. `on_print_complete` archiving a finished print) means `'completed'`. Anyone who genuinely wants to land an archive at `status='archived'` must now pass `print_data={'status': 'archived'}` explicitly so the intent is auditable at the call site. The `'archived'` enum value itself is kept for now — older rows might survive the migration on installs where m041 was interrupted, and the value is still referenced by the docs (Archive Status Badges in `archiving.md`); a future release can deprecate it once all installs have processed m041.

- **Legacy `print_archives.status='archived'` rows are hard-deleted on upgrade (`m041`), and `pending_uploads` is drained into the library.** Once Audits 1+2 (manual archive upload removal, VP queue redesign) and the m041 drain landed in 0.4.2 there is no production path that creates fresh `status='archived'` rows: the upload endpoints are gone, VP file ingestion goes through the library, and Audit-2 also removed the only path that produced new `pending_uploads` rows. The legacy `pending_uploads.py` action routes (`POST /archive-all`, `POST /{id}/archive`) still exist for one release as a backward-compatibility safety net for any external scripts, but after m041 drains every `status='pending'` row they have nothing to act on (the `Upload already processed` early return fires) and the frontend `PendingUploadsPanel` collapses to `null` because the list is empty — so no UI path can call them. Both routes plus the legacy table are scheduled for removal in 0.4.3 alongside the rest of the §Knock-on cleanup follow-up. Existing installations had piles of them — synthetic placeholders left over from the now-removed `POST /archives/upload`, the pre-redesign VP `_add_to_print_queue` / `_add_to_auto_queue` paths, and the pre-redesign `pending_uploads` approval routes. They were already filtered out of stats / list views by the codebase-wide `status != 'archived'` clause but cost disk + DB rows for nothing. New migration `m041` runs once at upgrade: **Part A** walks every `pending_uploads` row at `status='pending'`, hashes its temp file, dedups against existing `library_files.file_hash`, saves new bytes into the library (with metadata + thumbnail extraction matching the regular library upload path), records the resulting `library_files.id` in a new `pending_uploads.archived_to_library_id` audit-trail column, and flips the pending row to `status='archived'`. Pending rows whose temp file vanished are flipped to `'discarded'`. **Part B** explicitly NULLs `spool_usage_history.archive_id` for the soon-to-be-deleted archives (the FK has no `ON DELETE` clause so SQLite — which BamDude doesn't enable PRAGMA foreign_keys on — would otherwise leave dangling pointers), then hard-deletes every `print_archives.status='archived'` row in chunked DELETE statements. After the row delete, the migration walks the unique set of archive directories that pointed at deleted rows, checks if any *surviving* archive row still references a file under each directory (cross-printer file dedup means N rows can share one on-disk copy), and `shutil.rmtree`'s only the directories with zero remaining references — same ref-counted disk-cleanup contract `ArchiveService.delete_archive` already uses. **No PK renumbering** — `print_archives.id` is referenced by 6+ other tables (`print_queue` / `auto_queue` / `active_print_spoolman` / `library_files` / `project_bom` / `spool_usage_history`), and renumbering would require a multi-table rewrite for zero practical benefit. Idempotent — second run finds zero pending rows + zero archived rows. The frontend `PendingUploadsPanel` already returns `null` when the list is empty, so the panel disappears automatically post-migration; the `/pending-uploads/*` routes + table stay queryable for one release as a safety net for anyone scripting against the legacy API, then drop in 0.4.3.

- **Manual archive upload (`POST /api/v1/archives/upload` + `/upload-bulk`) is gone, along with the page-wide drag-and-drop on the Archives page and the `u` hotkey.** The endpoints created `print_archives` rows directly from operator-uploaded 3MFs — a back door that produced "archives" that were never actually printed by any printer (`printer_id=NULL`, `status='archived'`), polluting the print history with synthetic rows that the stats / cleanup / chain-of-custody paths then had to special-case. In the post-0.4.2 design archives = the print history of record (one row per physical print started by a real printer + the dispatcher), period. Operators who need to feed a 3MF into the system have three intent-matching paths: drag-drop the file onto a printer card on the Printers page (uploads to the library + immediately prints), upload via the File Manager modal (saves to the library for later use), or send via the slicer to a Virtual Printer (the redesign in §Audit-2 above lands the file in the library with a per-VP destination folder and queues it). Removed: `upload_archive` + `upload_archives_bulk` route handlers + their `archives:create` permission check, `api.uploadArchive` / `api.uploadArchivesBulk` client wrappers + `BulkUploadResult` type, `frontend/src/components/UploadModal.tsx` + its test file, the `'u'` keyboard shortcut and the page-wide drag-drop overlay on `ArchivesPage`, and `archives.releaseToUpload` + `archives.page.dropFilesHere` + `archives.page.only3mfSupported` i18n keys (en + uk). The `archives:create` enum value is **kept** in `core/permissions.py` so any existing `user_permissions` rows that grant it stay loadable — but it no longer gates any endpoint and is being phased out of the default Operator role in §Knock-on cleanup.
- **Upstream-ported archive auto-purge (per-row daily soft-delete) is gone.** It ran daily, soft-deleted any archive row whose own latest activity (`completed_at` / `started_at` / `created_at`) crossed the operator-configured threshold, and let the trash sweeper hard-delete it after the retention window. In a BamDude post-b1 world that turned out to be both **redundant and actively harmful**: redundant because the new "Auto-cleanup of stored 3MF files" prunes 3MF bytes per design chain (group by `effective_hash`, look at the newest activity across *all* chain siblings) which gives the same disk-reclaim effect with a much safer mental model; harmful because the upstream logic is per-row and BamDude's dispatcher creates a fresh archive row for every reprint, so a model printed weekly for two years would have ~70 of its earliest archive rows individually fall past a 365-day threshold even though the design was still hot — quietly destroying the print history (slicer-derived metadata, energy/cost rows, failure context) that BamDude exists to preserve. Removed: `archive_auto_purge_*` settings keys, `_maybe_run_auto_purge` / `purge_older_than` / `preview_purge` / `get_settings` / `set_settings` on `ArchivePurgeService`, the `/api/v1/archives/purge*` endpoints, the corresponding API client functions, the SettingsPage toggle + days input, and `archiveAutoPurge.{enableLabel,enableDescription,ageLabel,ageDescription}` i18n keys (en + uk). Kept: manual delete → trash → restore → empty-trash flow, the trash retention sweeper, and the `archive_trash_retention_days` setting — those serve the explicit "I want this row gone" workflow with a recovery window. Existing rows already in the trash bin are unaffected; the next sweeper tick still hard-deletes them after retention. Existing `archive_auto_purge_enabled` + `archive_auto_purge_days` rows in the `settings` table are now orphans (no reader) — left in place rather than running a settings-cleanup migration since they cost a row each and re-enabling the feature in a downgrade is impossible anyway.

### Changed

- **`source_content_hash` is now always populated; `m039` backfills the legacy NULLs.** The column on `print_archives` was originally only filled when the archived bytes differed from the chain root (i.e. patched library prints). Standalone rows and external prints with no chain ancestor were left with `NULL` source — the dedup layer compensated via `effective_hash = COALESCE(source_content_hash, content_hash)`, but the invariant "this column is the chain root" was blurry and every duplicate-detection query had to reach for `COALESCE`. New invariant: every row written by `archive_print()` / `attach_3mf_to_archive()` has a non-NULL `source_content_hash` — `chain_lookup` populates it from any pre-existing chain sibling, and standalone rows seed it with their own `content_hash` so they become the chain root for any future patched variant. New migration **m039** runs `UPDATE print_archives SET source_content_hash = content_hash WHERE source_content_hash IS NULL` so existing rows pick up the same invariant. `COALESCE` is kept everywhere as defence-in-depth (handles any hypothetical NULL row from a migration race that the migration runner shouldn't allow but might), but every read query effectively reads `source_content_hash` for new rows. Knock-on fix in `archives.py::list_archives` and `services/archive_comparison.py` and `services/archive.py::find_duplicates`: those code paths used to project archive rows to their *raw* `content_hash` for the page-side duplicate-badge filter / "similar by content" / "similar by name+hash" lookups, but the duplicate-detection service `get_duplicate_hashes_and_names` was already grouping SQL-side by effective hash — so any row whose `source_content_hash` differed from `content_hash` (i.e. every patched library print) was invisible to the badge logic. Now consistent: every page-side projection and SQL-side condition uses effective hash, so the "оригінальний друк" badge / `original_archive_id` link / detail-endpoint duplicates list all bind on `source_content_hash` (with `content_hash` only as a defence fallback for legacy NULL source rows). Net behaviour: a printer-A unpatched run + printer-B mesh-mode-disabled run of the same library file are now correctly grouped as duplicates of each other in the UI badge.
- **Cross-printer dedup for archive files on disk.** When archiving a print whose content hash matches an archive that already exists on *any* printer (previously only the same printer was checked), the new archive row reuses the existing on-disk path instead of writing another copy. `delete_archive` already ref-counts shared `file_path`s and only `rmtree`s when the last referencing row is hard-deleted, so deletion is safe. Net effect: printing the same file on N printers stores one copy on disk + N archive rows, instead of N copies. The on-disk path lives under whichever printer's folder created the file first, but no UI / API surface groups archives by file-system folder structure (everything indexes by row metadata), so this isn't user-visible. Saves both disk and the redundant `_copy_and_fsync` after download. Dedup is on **strict `content_hash` match**, not the chain-coalesce `effective_hash` — different patched variants of the same source (e.g. unpatched original `H1` from an external print on Printer A vs. mesh-mode-disabled patched bytes `H2` dispatched to Printer B) share `effective_hash=H1` for grouping but live as separate files on disk; aliasing between them would point one row's `content_hash=H2` at bytes `H1` and break every later `ZipFile` read. Chain linkage is preserved separately by the existing `chain_lookup` (sets `source_content_hash`), so the frontend still groups variants by `effective_hash` regardless of physical layout. Same dedup logic now also runs in `attach_3mf_to_archive` (background download-retry path) — when a fallback archive's 3MF lands later via the retry service and another printer already has the same bytes archived, the fallback row's `file_path` and `thumbnail_path` link to the existing copy instead of duplicating.
- **Archive 3MF cleanup: drift-mode schedule (24h since last run) replaces fixed midnight cron; manual runs now reset the auto-cycle.** The daily auto-tick used to fire at server-local 00:00 — predictable, but the operator's mental model didn't match the library auto-purge (which has always run "no more often than once per 24h after the previous successful run"). Two consequences: (1) admins had to keep two different schedules in their head depending on which bin they were thinking about, (2) clicking "Run now" from the Archives page modal didn't postpone the next auto-tick, so a manual cleanup at 23:00 followed by an auto-tick an hour later did the same scan twice. Both bins now share one pattern: tick every 15 min, run only when 24 h has elapsed since the previous successful run, and any successful run (auto OR manual) stamps `archive_3mf_cleanup_last_run` so the timer resets uniformly. Equivalently for the library: `library_trash_service.purge_older_than` now stamps `library_auto_purge_last_run` + records the in-memory result whether it was called by the auto-tick or by the admin's manual `/library/purge` endpoint. Side effect: removed `_seconds_until_next_local_midnight` from the archive cleanup service since wall-clock anchoring is gone — the schedule is now timezone-independent, surfaces neatly in the existing "Last run / Next run" cards, and an admin-triggered run won't have an auto-tick stepping on its heels minutes later.
- **Archive 3MF auto-cleanup: faster scan + tighter library-skip rule.** The daily sweep used to load every archive row with a non-empty `file_path` into Python memory just to group by design hash and find expired buckets — on installs with 10k+ archives that's a hot path that needlessly walks the entire table even when only a handful of buckets are actually expired. The plan phase now does a SQL-side `GROUP BY effective_hash HAVING MAX(activity_ts) < cutoff` first, then loads full rows only for the buckets that actually fell out of retention; healthy installs (most groups still hot) skip the row load entirely. Bug fix in the library-still-has-this-design skip rule: `or_(LibraryFile.file_hash.in_(hashes), LibraryFile.file_hash.in_(hashes))` carried two identical clauses (probably an editor-merge artifact) — same query, no functional difference, but the duplicate was removed and the search hash set was tightened to just `source_content_hash` (the canonical chain root after the always-fill change above; a patched archive's `content_hash` never matches any library file's hash anyway, so excluding it cuts the IN list size roughly in half on chained installs). Same retention semantics, same skip rules — just less wasted work.
- **Library auto-purge settings restyled to mirror the archive cleanup card.** File Manager card now opens with the auto-purge toggle (the toggle's expanded controls — days threshold + "include files never printed" checkbox — collapse out of view when disabled instead of staying visible-but-disabled), and the trash retention input is moved to the bottom so the order matches the archive settings card (auto first, trash retention last). The expanded block now also shows **Last run | Next run** cards (extracted into a shared `<LastNextRunCards>` component reused by archive cleanup) so admins can see "moved 5 file(s) to trash, 4 hours ago" + "in ~20 hours" without grepping logs. Backend bits: new `LibraryPurgeRunResult` dataclass, `library_trash_service.last_result` in-memory pointer, `library_trash_service.get_status()`, and a `GET /api/v1/library/trash/auto-purge/status` endpoint that returns `{enabled, days, include_never_printed, last_run, next_run_at}`. The in-memory result is restart-volatile (matches `archive_cleanup_service`) — when only the persistent `library_auto_purge_last_run` timestamp survives a restart, `last_run.moved` is reported as `-1` and the UI shows "count was lost on restart — see logs" instead of a misleading zero.
- **"Next run" hints in both auto-cleanup blocks now explain the first-fire delay symmetrically.** The hint under the Next-run card (Settings → File Manager for both archive 3MF cleanup + library auto-purge) now reads "Runs at most once per 24 h after the previous successful run. After enabling for the first time the loop ticks every 15 min — first auto-run lands within 15 min, then drifts on the 24 h cycle. A manual run resets the timer." Replaces two slightly-different one-liners ("Daily sweep runs at server-local midnight" on archive, "Auto-purge runs at most once per 24 h" on library) that didn't explain why the first run after enabling sometimes seems to wait — answer: the loop sleeps `TICK_INTERVAL_SECONDS=900s` *before* evaluating the gate, so first-tick lag is always 0–15 min. Same wording in en + uk.
- **Virtual printer ingestion: files now land in the library first, then the queue references them by `library_file_id` (no archive placeholder).** `print_queue` and `auto_queue` modes used to call `ArchiveService.archive_print(status='archived', ...)` to pre-create a synthetic archive row, then queue the file via `archive_id`. The dispatcher already knows how to print directly from a library file (`_run_print_library_file` creates the real archive at print-start with `status='printing'`), so the placeholder was dead intermediate state — every VP-uploaded file was creating two archive rows (one `archived` then one `printing`) and the placeholder was indistinguishable from a manually-archived legacy row in the stats / dedup paths. New flow: VP file ingestion routes through `_save_to_library()` first (returns the `LibraryFile`), then queues with `library_file_id=lib.id, archive_id=None`. The dispatcher's `_run_print_library_file` is the one and only producer of the archive row for a VP-originated print. Per-VP **library destination folder** picker added in the same change: new `virtual_printers.target_folder_id` column (FK → `library_folders`, `ON DELETE SET NULL`) shipped via migration **m040**, surfaced as a "Library destination folder" dropdown in the VP card (hidden for proxy mode since proxy bypasses VP file handling). Empty / NULL = library root. Read-only external folders are filtered out of the picker so operators can't pick a destination the VP can't actually write to. `_find_best_queue` switched from `(db, archive)` to `(db, sliced_model: str | None)` and its pending-time accounting now walks both archive- and library-file-backed queue rows so a queue full of library dispatches isn't seen as artificially empty. New helper `_extract_plate_id_from_metadata` reads the plate index out of `library_file.file_metadata['plates']` instead of reopening the 3MF a second time. Existing queue items keep working — the redesign only changes how *new* VP uploads create rows.

- **Archive page header "Cleanup" button now triggers the new per-design 3MF cleanup, not the removed per-row auto-purge.** The button + modal in the Archives page header (`PurgeArchivesModal`) used to call the now-deleted `previewArchivePurge` / `executeArchivePurge` endpoints to bulk-move old archive rows to trash; with that mechanism removed, the modal would have been dead code and broken at runtime. It is now wired to `getArchiveCleanupStatus` / `getArchiveCleanupPreview` / `runArchiveCleanup` (the same endpoints that back the Settings → File Manager block) so admins can run the stored-3MF cleanup on demand from the Archives page without leaving the screen. **Manual runs work even when auto-mode is disabled in Settings** — the `enabled` toggle only gates the daily auto-tick; this dialog (and the API endpoints behind it) honours the configured retention threshold regardless. When auto is off the modal shows a small amber hint "Auto-mode is off — only manual runs will fire" but the Run button stays active. **Per-run days override**: the modal now exposes a days input pre-seeded from the saved retention threshold (with a one-click "reset to {{days}}" link when the operator has dragged it). Both `GET /api/v1/archives/cleanup/preview` and `POST /api/v1/archives/cleanup/run` accept a `?days=N` query param (1–3650, clamped) for ad-hoc thresholds without persisting them to settings — same pattern as the library purge dialog. Button label is now "Cleanup 3MFs" / "Очистити 3MF" (was "Purge old" / "Очистити старі") and the modal copy was rewritten to match the new semantics: the cleanup keeps the archive row + thumbnail + metadata and only deletes the 3MF bytes, so reprint / queue / download buttons disable on affected rows but the history stays for analytics.

### Fixed

- **Restart-recovery now adopts in-flight prints whose 3MF was patched.** Library / reprint dispatch with patches (mesh-mode-fast-check off, per-plate G-code injection) used to record `content_hash = source_content_hash = library_files.file_hash` (i.e. the unpatched library bytes) on the archive row, while the FTP step uploaded patched bytes to the printer's SD card. After a BamDude restart mid-print, `on_print_start`'s post-download adoption block downloaded the printer's copy (patched), hashed it, and looked for `content_hash == temp_hash` — never matching, so it created a fallback archive instead of adopting the in-flight one. Six prints with the same patch set + a restart = six orphan fallback rows + six lost references to the original dispatch context (created_by, project, library link, swap intent, queue id). New `archive_print(..., dispatched_file=...)` parameter splits the role of the old single-file argument: `source_file` carries chain identity (display name / suffix / `source_content_hash` derivation) and `dispatched_file` is the post-patcher tempfile that goes to FTP. `content_hash` is now SHA256(`dispatched_file`) and the archive folder copy is `dispatched_file` too, so on-disk + on-printer + DB hash all agree. `source_content_hash` stays as the chain root (explicit param, or computed from `source_file` via `chain_lookup`). Default `dispatched_file=None` keeps every legacy single-file caller (external prints, VP file_manager save, library upload without patching) on the original code path with no behavioural change. Cross-printer file dedup keys on `content_hash` and patches are deterministic, so 6 prints with the same patches still share one on-disk archive copy of the patched bytes — no extra disk. Three new regression tests in `test_archive_dispatched_file.py` cover (1) patched dispatched bytes → split hashes, (2) no `dispatched_file` → legacy single-hash path, (3) cross-printer dedup with the new param.

- **Reprint no longer mutates the source archive — it creates a new row.** Clicking "Reprint" on a `failed` (or `cancelled`, or even `completed`) archive used to feed the source archive's `id` into the dispatcher; `_run_reprint_archive` then registered that `id` as the expected print, and `on_print_start` unconditionally flipped the row's `status='failed'` → `'printing'` and cleared `completed_at`. The historical failure / completion record was silently destroyed and the new run shared a row with the old one — no way to look back and see "this design failed once on printer A before succeeding on printer B." Reprinting the same source on N printers compressed N runs into one mutating row. Same principle as the 0.4.2 archive-table audit cycle: archives are the **print history of record**, one row per physical print run, no exceptions. Fix mirrors the library-file dispatch path (`_run_print_library_file`): the dispatcher now calls `archive_print(source_file=archive.file_path, source_content_hash=source.source_content_hash, library_file_id=source.library_file_id, plate_index=source.plate_index, created_by_id=requested_by_user_id, print_data={'status': 'printing'}, ...)` to mint a fresh row that inherits chain identity from the source (so duplicate-detection still groups them) but has its own id, status, started_at, energy tracking, and creator attribution. The source row is never read for write — it stays in the history with its original terminal state intact. Cross-printer file dedup inside `archive_print` already reuses the source's on-disk `file_path` whenever `content_hash` matches (typical when the reprint applies the same patches), so the new row costs one DB row + zero extra disk. Queue-driven reprints additionally re-point `print_queue.archive_id` at the new row (mirroring the library-file path) so the queue UI tracks the actual run instead of the historical source. Same patcher cleanup, same `register_swap_config`, same start_print verification — only the archive identity changes. `register_expected_print` is now called with the new row's id, and on_print_start fills in `started_at` + energy_start_kwh on the new row (it's born `status='printing'` so no destructive status flip happens regardless).

- **Auto-cleanup result count survives a server restart.** Both auto-cleanups (archive 3MF + library auto-purge) used to lose the in-memory `LastResult` on process restart and synthesize a sentinel `archives_cleared = -1` / `moved = -1` from the persisted timestamp alone — UI then showed "count was lost on restart — see logs" even when the actual count was zero or a small known number. New persistent settings keys store the count alongside the timestamp: `archive_3mf_cleanup_last_archives_cleared` + `archive_3mf_cleanup_last_bytes_freed` (archive) and `library_auto_purge_last_moved` (library). Stamped uniformly by both auto-tick and manual paths. After restart the synthesised result picks up the real count from settings; the `-1` sentinel only fires for legacy stamps that pre-date these keys (i.e. one transitional restart per install). Net effect: a routine restart no longer makes a perfectly-fine 0-result auto-tick look broken.
- **Archive cleanup status card no longer shows a stale "Next run: —" for up to a minute after enabling auto-mode.** The Settings page persists archive 3MF retention via the global settings save mutation (debounced 500 ms + global save), but `updateMutation.onSuccess` didn't invalidate `archive-cleanup-status` — that query polls on its own 60 s interval, so the Last/Next run cards kept rendering the pre-toggle snapshot (`enabled=false → next_run_at=null → "—"`) for up to ~60 s after the user flipped auto-mode on. Now invalidates `archive-cleanup-status` + `archive-cleanup-preview` alongside the existing invalidations on every settings save. Library auto-purge already had this — the library trash mutation invalidates `library-auto-purge-status` on success since b3.
- **`<LastNextRunCards>` shows an optimistic "in ~15 min" instead of a dash during the brief race between toggle-on and the first refetched status.** Even with the invalidation above, there's a ~700 ms-1 s gap between flipping the toggle (debounce 500 ms → settings PUT roundtrip → status invalidate → refetch) where the local toggle says ON but the cached status data still shows `next_run_at=null`. The component now detects `autoEnabled === true && nextRunAt === null` and renders `"in ~{TICK_INTERVAL_MIN} min"` — same value the backend returns once the refetch lands. Bonus: the same component now shows `t('autoModeOff')` ("auto-mode is off" / "авто-режим вимкнено") in the Next-run card when the toggle is genuinely off (`autoEnabled === false && nextRunAt === null`) instead of the bare `—` placeholder; `autoEnabled` is a new optional prop wired from both call sites. Removes a confusing UX where a user enabled the toggle, saw "—" for both cards, and assumed the feature was broken.

- **Library-flow now archives the unpatched original on disk (not the patched upload).** Printing a library file with `mesh_mode_fast_check=False` (or per-plate G-code injection) used to drop the *patched* bytes into the archive directory — visible in `data/archive/<printer>/<timestamp>_<name>/` as a 3MF whose size and hash differed from the library original. The reprint UI lets the user re-pick mesh-mode / injection on every run, so storing patched bytes meant new toggles would silently compose on top of already-baked patches. The hash columns were already correct (`source_content_hash` = library original, `content_hash` = patched), but the on-disk file disagreed with the design intent. Fix: `_run_print_library_file` now passes the unpatched library file as `source_file=` to `archive_print()` while FTP upload continues to send the patched copy. `applied_patches` keeps recording what was applied to the bytes that actually went to the printer (informational only — never used to infer disk-file state). Library archives now end up with `content_hash == source_content_hash == library_files.file_hash`; chain-of-custody consumers use `COALESCE(source, content)` so this is a no-op for grouping while letting cross-printer file dedup share an on-disk copy across runs that picked different patches. Re-printing from the archive page (archive-flow) is unchanged — it reads `archive.file_path` (now also unpatched for library-originated archives) and re-runs the patcher per-job, exactly as the design intended.
- **Reprinting a file from the printer screen no longer overwrites prior history.** When a user reprinted a previously-completed file from the printer's own screen (across one or many printers), `on_print_start`'s post-download content-hash adoption block was matching the historical archive by `content_hash` *regardless of status* and flipping it from `completed` (or any terminal status) back to `printing` — clearing `completed_at`, deleting the original print's outcome, and never creating a new archive row for the new run. Doing this on N printers at once destroyed N rows of history. Adoption now requires `status="printing" AND completed_at IS NULL` — terminal rows are left alone, and the new print gets its own fresh archive. Mid-print recovery (BamDude restart while a print was actively running) keeps working because the in-flight row's status is still `printing`.
- **Cross-printer race when multiple printers download the same-named 3MF concurrently.** When a user kicked off the same file on two printers from the slicer at once, both `on_print_start` flows raced into the same shared temp path (`archive_files/temp/<filename>.gcode.3mf`) — both opened it with `"wb"`, truncated each other's output, and the resulting bytes were a sparse / corrupt 3MF. On Linux this typically produced two archives both pointing at garbage; on Windows the second `open` failed with a sharing violation and that printer fell through to the no-3MF fallback. Fix: temp downloads now go into a per-printer subdir (`archive_files/temp/<printer_id>/<filename>.gcode.3mf`) so two printers can't collide on the same path. `archive_download_retry` and the on-demand `try_download_3mf` helper both pick this up automatically — no caller changes needed.
- **Library trash hard-delete + sweeper now explicitly NULL out the FK on related archive rows on SQLite.** `print_archives.library_file_id` is declared `ON DELETE SET NULL` in the model + every migration that touches it, but SQLite only enforces FK actions when `PRAGMA foreign_keys = ON` is set on each connection — which BamDude deliberately does not (turning it on globally would require a separate audit of every other `ON DELETE` clause in the schema). On Postgres the FK fired correctly; on SQLite installs (the default) two of the four library hard-delete paths — `library_trash_service._sweep` (after retention elapses) and `library_trash_service.hard_delete_now` ("Empty trash" / "Delete now") — left related archive rows with **stale `library_file_id` pointers** to now-deleted library rows. The user-visible symptom is subtle (later joins return NULL, the cleanup library-skip rule still treats the design as "library has it gone" and proceeds correctly), but the inconsistency would have surfaced any time something assumed `library_file_id IS NULL ⇔ never had a library link`. The two route-level library-delete paths (`api/routes/library.py` external-file delete + bulk delete) already did an explicit `UPDATE PrintArchive SET library_file_id = NULL WHERE library_file_id IN (deleted_ids)` before the row delete; the trash sweeper + `hard_delete_now` now do the same. Behaviour now matches across both DBs and across all four entry points: archive rows survive a library hard-delete with the FK column blanked, **never cascade-deleted**.
- **Same-second same-name archive folder collision now appends a suffix.** `archive_print` and `attach_3mf_to_archive` build the archive folder name as `YYYYMMDD_HHMMSS_<stem>` — per-second granularity. In practice a single printer can't run two prints in the same second so this was a theoretical-only risk, but the previous `mkdir(parents=True, exist_ok=True)` would have silently re-used a colliding folder and overwritten the existing 3MF. New code probes for an existing folder and appends `_2`, `_3`, … to the name until a fresh path is found, then `mkdir(parents=True)` without `exist_ok` so any future race fails loud.

### Archive 3MF auto-cleanup

New optional retention policy for the archive directory: when a design hasn't been printed for N days, its `.3mf` file is deleted from disk. The archive row itself stays — thumbnails, filament usage, costs, notes, energy data, project links, and all other metadata are preserved. Re-printing an old design after cleanup just shows "3MF unavailable" until you re-upload from your library or slicer (same UX as fallback archives that never had a 3MF).

**Settings → Print → Archive Settings → "Auto-cleanup of stored 3MF files"** — toggle, retention input (default 30 days, minimum 1), live preview of what would be cleaned right now (`X archives in Y groups, Z MB`), last/next run status, and a "Run cleanup now" button with confirmation.

**Behaviour:**

- **Group by design hash, not per-archive row.** Reprints of the same model share a `COALESCE(source_content_hash, content_hash)` group; the cutoff is evaluated against the newest print across the whole group, so reprinting a model 3 days ago keeps every archive copy of that design — even rows from months back. When the design *is* eligible, every copy in the group has its `file_path` blanked together (consistent UI), and the unique physical file on disk is removed once.
- **Skip rules.** Skipped if any row in the group is `status='printing'`, an active queue item references it (pending / printing / paused), or the originating `library_files` row still exists with a matching hash — the library is the operator's deliberate source of truth, no need to wipe an archive copy when it's still on the source path. If any rule fires, the *whole* group is skipped.
- **Daily sweep at server-local midnight.** `archive_cleanup_service.py` runs the loop; the gate is the `archive_3mf_retention_enabled` setting, so toggling on/off takes effect without a restart. Manual "Run now" honours the same skip rules.
- **What's deleted.** The `.3mf` file + its `.gcode.md5` sidecars + the per-archive directory if it ends up empty. The thumbnail PNG and the archive row stay intact.

API: `GET /archives/cleanup/status`, `GET /archives/cleanup/preview`, `POST /archives/cleanup/run` (preview/status need `archives:read`, run needs `archives:delete_all`).

### Parallel dispatch with startup-lock

Background dispatch now runs jobs **in parallel** across printers — sending a print to two or more printers at the same time no longer queues them one-after-another. The previous full-serialization gate (one job in flight across the farm) was added to dodge a "database is locked" deadlock, but it serialized the slow FTP / start_print phases too, which on a busy farm meant operators waited several minutes before the second printer even started receiving its file.

**Mechanic** — replaced the global "one at a time" gate in `_dispatcher_loop` with a `self._startup_lock` (asyncio.Lock) that wraps **only** each job's DB-write critical section: `archive_print` (the heavy `INSERT INTO print_archives`) plus queue-item linking, with an explicit `commit()` at the end. The lock releases right after that commit, so the FTP upload + MQTT `start_print` + post-write phases all run concurrently with the next job's setup. This eliminates the original deadlock root cause (an open write-txn left over from `db.flush()` was held throughout FTP, blocking the next job's INSERT for longer than SQLite's 15s busy timeout) without giving up parallelism on the long phases.

**Affected paths** — `_run_reprint_archive` and `_run_print_library_file`. Both wrap their DB-write phase in `acquire()` / `try / finally release()`. Reprint's lock window is just the optional `swap_macro_events_pending` pre-stamp commit; library-file's lock window covers archive INSERT + queue linking + commit. Post-FTP follow-up writes (swap-pending checklist tick, optional library-file cleanup, final commit) stay outside the lock — they're short and serialize naturally via SQLite's WAL + busy-timeout.

Two new fields are now extracted from each 3MF's `Metadata/project_settings.config` and stored alongside the existing print metadata:

Both fields land in `library_files.file_metadata` (uploads + re-parses) and `print_archives.extra_data` (new prints + recovered fallback archives). Forward path is automatic — the existing `ThreeMFParser` populates them on every parse.

### Fixes

- **Frontend recovers correctly from a long-backgrounded tab.** When a browser tab sat in the background for ~30 minutes (other tab focused, or another window in front), the WebSocket connection often stayed open in name only — the OS / browser had silently torn down the underlying socket without firing `onclose`. Bringing the BamDude tab forward then showed stale React-Query cache for several seconds while a new dispatch toast occasionally rendered "live" with already-finished events as the cache caught up. Three reinforcing changes:
  - **Pong-timeout watchdog** — every 30 s ping now arms a 10 s pong-timeout. If the server doesn't reply within 10 s the socket is considered dead and force-closed, kicking the standard reconnect path. Catches silent socket teardown that browsers used to hide.
  - **Visibility-sync** — `document.visibilitychange='visible'` triggers (a) `queryClient.invalidateQueries({ refetchType: 'all' })` so even non-active queries (cached pages the user is about to navigate to) refresh, and (b) an on-demand ping that reuses the same pong-timeout watchdog above. If the socket is alive, the pong arrives, nothing else happens — quick Alt+Tab no longer churns the connection. If the pong doesn't arrive within 10 s, the watchdog fires the standard close→reconnect path. The on-window-focus refetch built into TanStack Query already covered the active queries, but inactive queries (e.g. the Queue cache while you're reading Archives) used to stay stale indefinitely.
  - **"Reconnecting…" toast** — when the WS has been disconnected for >2 s, BamDude shows a persistent warning toast through the standard `ToastProvider` (same look as macro-executed / plate-not-empty / dispatch-progress notifications). 2 s grace prevents flicker during sub-second reconnects. Auto-dismisses on reconnect. Source of truth is a new `ConnectionContext` so any component can watch `isConnected` if needed.
- **`sqlite3.OperationalError: database is locked` on parallel dispatches.** Sending two prints to two different printers back-to-back could fail the second one mid-FTP — both jobs raced on `INSERT INTO print_archives` while one of them held an open write transaction through the entire FTP upload window, so SQLite's single-writer semantics gave up after the busy-timeout. Fixed by the new parallel-dispatch flow with a per-job startup-lock around the DB-write phase (see "Parallel dispatch with startup-lock" above). The interim "one job at a time across the whole farm" gate that landed earlier in the 0.4.1 cycle is gone — parallel sends now work without the deadlock.
- **Skip-objects state no longer leaks between prints.** `_load_objects_from_archive` previously left `printer.state.printable_objects` untouched when the parser returned an empty dict (e.g. Orca-sliced 3MFs where `support_skip_objects` was off — the runtime label_object markers in gcode work, but `slice_info.config` carries no `<object>` entries our parser can use). On the next print of a different file, the stale dict from the previous job remained in state — the skip-objects modal would either show object IDs that don't exist in the new gcode (firmware rejects the resulting `M623`), or silently misreport the count. State is now reset unconditionally as soon as the function is called for a given archive — if parsing yields nothing, the modal stays hidden, which is the truthful UI for "we don't know the object list for this print".
- **Skip-objects button now requires slicer-side support flags.** A 3MF can carry `printable_objects` metadata (so the count looks right) yet still be unusable for per-object skipping if the slicer profile didn't enable it — clicking the button would send `M623` for object IDs the firmware doesn't actually have markers for, and the firmware would reject. The button now also requires `gcode_label_objects=true` AND `exclude_object=true` in the source 3MF's `Metadata/project_settings.config` (extracted by the parser into `archive.extra_data`, surfaced as `skip_objects_supported` on the printer status payload). Affects three call-sites: `PrintersPage.tsx` (header button + disabled-state styling), `EmbeddedCameraViewer.tsx`, `CameraPage.tsx`. Old archives without the m022 backfill (3MF deleted before backfill ran) fail this gate cleanly — button hidden, no firmware-reject surprises.
- **Skip-objects button no longer "dies" 5 minutes into a print.** Per-printer `mqtt_connection_timeout` (default 300 s for printers imported from Bambuddy) periodically swaps `BambuMQTTClient` for a new instance via `ensure_fresh_connection`. The fresh state had `printable_objects={}` / `skip_objects_supported=False` until `on_print_start` re-fired and reloaded — but the duplicate-guard at the top of `on_print_start` (which exists to prevent fallback-archive duplicates on MQTT replays) returned early *before* the reload, so the skip-objects button went dark every ~5 min mid-print. Server restart "fixed" the symptom because it also wiped `_active_prints` (the dict the duplicate guard reads), letting the next on_print_start take the full path. The duplicate-guard branch now also re-runs `_load_objects_from_archive(active_archive)` so the freshly-created MQTT client state is repopulated alongside skipping the duplicate notification path.
- **Skip-objects state for printer-started fallback prints.** When a print starts directly from the printer screen / Send-to-Printer and BamDude can only fetch the 3MF on a later retry hook (`archive_download_retry`), the new `load_objects_from_archive_into_state` helper now runs as part of the successful attach — the skip-objects modal sees the object list as soon as the recovery completes, instead of waiting for the next BamDude restart or staying hidden for the whole print.
- **Dispatch toast progress bar no longer "sticks" past upload phase.** After upload finished `_set_active_upload_progress(job, 1, 1)` left `upload_bytes=upload_total_bytes=1` on the active dispatch state. Subsequent phases (`Running swap start macro…`, `Starting print on…`) updated the toast text but the progress bar kept rendering at 100% because nothing cleared the byte fields — looked like the file was still uploading while the swap macro was actually running. `_set_active_message` now resets both byte counters when a new phase begins; if a new upload phase starts, its first progress callback repopulates them.
- **No more 401 burst in the browser console after a long-idle tab.** Sliding-session refresh is reactive — when an access token expires the next request gets `401 Token has expired`, the client calls `/auth/refresh`, then retries. The user never gets kicked out, but the browser network panel logs every parallel 401 (20-40 of them when 4-5 React-Query keys re-fetch on tab-refocus). `client.ts` now decodes the JWT's `exp` claim and arms a one-shot timer to call `/auth/refresh` ~60 s before expiry, plus a near-expiry pre-flight check at the top of `request()` that awaits the same coalesced refresh promise when a request is about to leave with a near-dead token. Result: in the common "tab sat in background, came back" case the access token is already fresh when the visibility-sync invalidates queries — no 401s leave the browser, console stays clean. The reactive path stays as a fallback for direct-fetch call-sites (binary / streaming endpoints) and for failure modes where the proactive refresh didn't run (browser throttled `setTimeout`, refresh cookie expired). Decode failure (non-JWT or malformed token) safely degrades to the old reactive flow.
- **`print_finished` macro event has a translated label** in the macro settings table. The event landed in 0.4.0 (chamber-light symmetry, see "Auto-light-off superseded by macros" below) but `settings.macroEvents.print_finished` was missing from both `en.ts` and `uk.ts`, so the dropdown rendered the raw key. Fixed in both locales.
- **P2S printer card / archive thumbnail now shows the correct render.** `getPrinterImage()` mapped `p2s` to `/img/printers/p1s.png` because the project hadn't shipped a P2S asset yet — every P2S in the dashboard, queue widget, archive grid, and Telegram-bot status header rendered as a P1S. The 320 × 320 P2S render (sourced from Bambu Lab's wiki) is now bundled at `static/img/printers/p2s.png` and the mapping points at it. Existing P2S printers pick up the new image on next page load with no DB or config touch.
- **Swap `change_table` macro now survives a backend restart mid-print.** `_active_swap_config` was an in-memory dict — restarting BamDude between print-start and print-complete wiped the swap intent, and `on_print_complete` had nothing to act on (no `[SWAP] Running change_table` log entry). The intent is now also persisted to `print_archives.extra_data["swap_macro_events_pending"]` as a real checklist: every event the dispatch plans to fire is added at archive creation, and each event is removed from the list at the moment the corresponding macro fires successfully (`swap_mode_start` → removed by dispatch right after the macro returns; `swap_mode_change_table` → removed by `on_print_complete` right after the macro returns). When the list becomes empty the key drops entirely for clean state. Restart in the middle of a print: `on_print_complete` reads the remaining pending list and only fires what's still in it — a re-trigger of `on_print_complete` (MQTT replay, reconnect flap) finds the event already removed and does nothing. For library-file dispatches the initial marker is folded into `archive_print()`'s INSERT (same statement that creates the row, so no extra writer); for reprint dispatches the existing row is updated in the already-open dispatch session before FTP upload and any runtime-tracker contention. The earlier post-`start_print` standalone UPDATE was scrapped — it raced the runtime-tracker `UPDATE printers …` writer and timed out under SQLite's `busy_timeout`.

### Queue / Dispatch (post-b1, follow-up)

- **Queue history moves into archives — live queue only tracks what's unfinished** (migration `m019`). Three new columns on `print_archives`: `queue_id` (FK to `printer_queues.id`, `SET NULL` on delete), `batch_id` (VARCHAR(36)), `error_message` (TEXT, verbose-diagnostic twin of `failure_reason`). ### Queue / Dispatch (post-b1)

- **Single dispatch layer — kills the double-archive race.** Before: two orchestrators could both call `archive_print()` for the same (printer, library_file) if their timing collided — scheduler's `print_scheduler._dispatch_item` (queue-item branch) and `background_dispatch._run_print_library_file` (direct "Print Now"). Result: two archive rows for one physical print, one carrying `library_file_id`/`project_id`/`created_by_id` but stuck at `status='archived'`, the other getting `status='printing'` from `on_print_start` hash-adoption but with most metadata `NULL`. Now: the scheduler's `_start_print` is a thin wrapper that delegates the full patch → archive → FTP → register_expected → MQTT-start pipeline to `background_dispatch.run_from_queue_item()`. Only one dispatch runs per queue item; its archive row carries all the metadata and is wired to the queue item via `PrintQueueItem.archive_id` in the same transaction as archive creation. Removes ~250 lines of duplicated code from the scheduler. Stagger / is-idle / auto-drying / cancel-handling stays in scheduler — it's still the queue orchestrator, just stops executing prints itself.
- **Race-dup archive cleanup** — `scripts/merge_race_duplicate_archives.py`. Finds archive pairs with same `printer_id + content_hash + file_path` created within 60 s and asymmetric `library_file_id` (the classic race signature), picks a survivor row (queue-referenced first, then furthest along lifecycle, then started_at-set, then newer id), backfills every null-but-present field from the orphan, re-points any dependent `print_queue.archive_id`, deletes the orphan. `--dry-run` by default, `--apply` to commit, `--loose` prints broader candidates (same-hash within 5 min, at least one `archived`) for manual review without touching them.

### Authentication (post-b1)

- **Sliding-session refresh tokens** (§18.14). Access-token TTL drops from 24 h to 1 h, but no one has to re-log in every hour — a new rotating refresh cookie transparently refreshes the access JWT in the background. Leaving a tab open for a week keeps you signed in; closing the browser without the Remember-me checkbox logs you out (12 h DB cap regardless). **Remember me for 30 days** checkbox on the login form opts into persistent sessions — cookie survives browser restarts. Frontend retries failed 401s transparently through /auth/refresh, with promise-coalescing so a wave of parallel queries spawns exactly one refresh call. ### v0.2.3.2 port — safety, queue reliability, AMS UX

### Safety

- **Bed-jog "Home Z" no longer crashes the bed into the toolhead on H2C / H2D / H2S / X1** (upstream #1052). The bed-jog dialog's button sent a bare `G28 Z` over `gcode_line`. On Bambu printers where the Z endstop is at the top (bed moves UP into it), `G28 Z` skips the toolhead-park step that a full `G28` runs first, so the bed rises at full speed with nothing getting out of the way. The reporter only escaped damage because the toolhead happened to be parked on the purge chute. The `/printers/{id}/home-axes` endpoint and `BambuClient.home_axes()` now always send bare `G28` regardless of the `axes` argument, triggering the firmware's safe multi-step routine (park toolhead → home XY → home Z). Frontend button retitled "Auto Home" with dialog copy updated in en + uk so users aren't surprised when X/Y motion happens before Z.
- **"Not homed" warning no longer re-prompts after Auto Home** (upstream #1052 follow-up, `1de4e409`). Clicking "Auto Home" sent the G28 and closed the modal but never set the session-scoped `bamdude.bedJog.warned.<printerId>` flag (only "Move anyway" did), so the next jog click in the same session re-opened the warning — even though the printer was now homed. The `homeAxes` mutation's `onSuccess` handler now flips the same flag. Warning still fires once per printer per session as a safety guard.

### Queue Reliability

- **H2D false-reprint fix** (upstream #1078). The scheduler watchdog reverted queue items to `pending` at 45 s if `gcode_state` hadn't transitioned — on H2D Pro firmware 01.01.00.00 the state stays at `FINISH` for 48–55 s after the printer accepts `project_file` but before `PREPARE`, so the watchdog kept reverting items the printer had already started physically printing, and the next scheduler tick re-dispatched them as a reprint. `_watchdog_print_start` now also exits early when `subtask_id` advances past the pre-dispatch value (the printer echoes the `submission_id` we minted back in its next `push_status` — the signal lands long before `gcode_state` does on slow firmware). Timeout bumped 45 s → 90 s as belt-and-braces for printers that neither flip state nor echo `subtask_id` inside the window.
- **Batch double-dispatch fix** (upstream `950286ad` / v0.2.3.2 re-release). When an ASAP print with `quantity > 1` landed on an H2D or P1-series, the scheduler's next tick saw MQTT `IDLE` (state lagging the accepted command) and dispatched the second batch item onto the already-running printer — surfaced via the "BUG: Multiple queue items in 'printing' status" log. `check_queue` now seeds `busy_printers` from `PrinterQueue.status='printing'` before iterating, so any printer with an outstanding dispatch is excluded regardless of what MQTT currently reports. Also catches external / direct prints that don't have a corresponding queue item row.
- **Clear-Plate button latency fix** (upstream #939 follow-up, `4e86e8cb`). The card-level "Mark plate as cleared" button appeared 30 s – 5 min late because the WebSocket `printer_status` payload didn't carry `awaiting_plate_clear` — the frontend merged the stale `false` until the next 30 s REST poll refreshed it. `printer_state_to_dict` now emits the flag from the authoritative in-memory + DB-backed accessor so the button appears within ~100 ms of the gate being armed.

### Printer UI

- **Per-card plate name for multi-plate prints** (upstream #881, `fa1c46d9`). When two printers run different plates of the same multi-plate 3MF, the Printers page cards now show the actual plate name (or "Plate N" fallback) instead of just the 3MF filename — no more guessing which printer is on which plate. Sourced via `current_archive_id` (resolved from MQTT `subtask_id`) + `current_plate_id` (parsed from `gcode_file` path). Plate transitions reflect within ~100 ms via WebSocket push; single-plate prints stay clean.

### AMS / Filament

- **AMS-HT slot shows saved preset after Configure** (upstream #1053, `87a5aa36`). `GET /printers/{id}/slot-presets` keyed AMS-HT entries at `ams_id * 4 + tray_id = 512`, but frontend lookups use `ams_id` directly for HT (single-slot units share their global tray id with the unit id). - **Configure AMS slot preserves PFUS preset id** (upstream #1053, `28f80f94`). When Bambu Cloud returned `filament_id=null` for a user preset that inherits from a generic base (e.g. "Sting3D ABS" on "Generic ABS @BBL H2D"), the modal fell back to `convertToTrayInfoIdx(base_id)` — which collapsed the slot's filament identity to the generic's `filament_id`. OrcaSlicer / BambuStudio Sync Filaments then resolved the slot to "Generic ABS" and the custom preset never appeared on the printer LCD. - **Assign Spool surfaces manually-added inventory spools** (upstream #1047, `bc895dff`). Three related AMS-UX fixes. (1) Empty-slot hover card lost its Configure / Assign actions after a firmware-reset because a prior gate keyed off `tray?.state === 10`, but real firmware reports state=9 (or omits state entirely) after a user-initiated reset — gate removed (the hover card only renders for empty tray types anyway). (2) The spool filter in Assign Spool required exact `slicer_filament_name` equality, hiding every manually-added inventory spool that doesn't populate the field — filter now prefers exact slicer-profile match when both sides have one, and falls back to partial material overlap either direction ("PLA" spool shows up for a "PLA Basic" slot). (3) The profile-mismatch warning dialog fired on every Generic spool because slicer profile names carry an `@printer` qualifier while the tray stores the bare base name — both the filter and `checkProfileMatch` now strip everything from `@` onward before comparing.
- **Malformed rgba no longer bricks the Filaments page** (upstream #1055, `bf5135cb`). A single legacy spool with a 7-char rgba (`FFFFFFF`, one `F` missing) caused `GET /api/v1/inventory/spools` to 500 with a pydantic validation error, leaving the Filaments page blank and "Add Spool" silently failing. ### Queue Reliability

- **Signal-based MQTT zombie detection** (#887). The existing `ensure_fresh_connection` was a time-based preflight; it couldn't tell the difference between a quiet but healthy session and a half-broken one where our publishes silently drop. New mechanism: `BambuMQTTClient.force_reconnect_stale_session(reason)` closes the MQTT socket so paho auto-reconnects with a fresh session. Triggered by three signal-based paths: `ams_filament_setting` unanswered 2×10 s, dev-mode probe unanswered 2× (already landed, now routed through the shared helper), and `_verify_print_response` in `background_dispatch` after an unacked `start_print`. The preflight is kept for Telegram-handler UX predictability.
- **`_watchdog_print_start` in print_scheduler** (#967). Queue item marked "printing" after MQTT publish that the printer never processes used to stay stuck forever. The watchdog polls for a state transition and, if none arrives within 45 s, reverts the item to `pending` and force-reconnects the MQTT session. Swap-mode-safe: if `swap_mode_start` already physically moved the plate, the revert is skipped (we don't re-fire the macro on retry) and the operator is asked to intervene instead.
- **`PrintArchive.subtask_id` + resume pre-check** (#972). The printer's `submission_id` (the new unique-per-invocation identifier in `start_print`) is captured as `subtask_id` on `print_archives`. `on_print_start` consults it as a pre-match before the name-based lookup, so a backend restart mid-print adopts the right row even when two archives share a `print_name`. Also backfills `subtask_id` onto rows that matched by name on older runs. Unlike upstream's companion revive-from-cancelled path we deliberately do NOT un-cancel terminal rows — BamDude removed upstream's 4-hour stale-cancel heuristic long ago (it killed legitimate long prints), so there's nothing to revive.
- **`printers.awaiting_plate_clear` persisted column** (#961). The plate-clear gate used to live in an in-memory set; an Auto Off power cycle dropped it and the queue auto-dispatched onto a dirty plate. Now persisted to DB + rehydrated at startup. Semantics inverted: presence means **blocked on user confirmation** (absence = clear to dispatch). Armed at end of `on_print_complete` when `require_plate_clear` is true and no swap path already cleared the plate. Frontend: `plateCleared` prop/field renamed to `awaitingPlateClear`; `clear_plate` endpoint now also accepts `IDLE` state so post-Auto-Off recovery works. Upstream's parallel move of `require_plate_clear` to a global setting is deliberately NOT ported — BamDude keeps the per-printer column with default `true`.
- **`submission_id` on `start_print`** (#1011). Replaces the hardcoded `project_id / subtask_id / task_id = "0"` with a millisecond-timestamp UUID. Third-party MQTT observers (OctoEverywhere etc.) used to treat every reprint as a continuation of the prior task because `task_id` collided.

- **`FileNotOnPrinterError` short-circuit on 550** (#972). ### Authentication / 2FA / SSO

- **Two-factor authentication — TOTP + Email OTP + backup codes** (§18.1 / upstream #933). Users can enrol an authenticator app (Google Authenticator, Aegis, Authy etc.) via QR code or manual secret entry, optionally add email-OTP as a fallback, and receive ten single-use backup codes for account recovery. Login splits into a two-step flow: password → pre-auth token → 2FA code (TOTP/email/backup) → JWT. A password change or admin reset invalidates all outstanding sessions for that user (password_changed_at + JWT iat freshness). Backup codes are pbkdf2-hashed at rest; the TOTP secret is encrypted with Fernet when `MFA_ENCRYPTION_KEY` is set (optional — plaintext storage with a warning otherwise, plus prefix-versioned so you can enable encryption later without re-enrolling).
- **OIDC / SSO login** (§18.2). Admins can wire any OpenID Connect provider (Authentik, Keycloak, Pocket-ID, Google Workspace, etc.) and users see provider buttons on the login page alongside username/password. The authorization-code flow uses PKCE S256 + state + nonce + the discovery endpoint, and the callback atomically consumes its state token so replays can't succeed. Newly-linked accounts can either be auto-created in the Viewers group or linked to an existing local user by email (opt-in per provider, default off — stops a compromised IdP from hijacking local accounts). SSRF guards on the issuer URL reject non-HTTPS, loopback, private-network, and link-local targets.
- **JWT revocation + session freshness** (§18.4). Every access token now carries `jti` (per-token revocation id) and `iat` (issued-at). `/auth/logout` revokes the caller's jti until its original expiry; the revocation list is consulted on every request. Password changes stamp `user.password_changed_at`, and any token with `iat` older than that value is rejected as stale so a password reset actually logs out every existing session. Access-token TTL dropped from 7 days to 24 hours (reduces the window between a revoked-but-unnoticed token and its natural expiry). Slicer download and camera stream tokens moved from in-memory dicts to the `auth_ephemeral_tokens` table — survives server restarts and works cleanly under multi-worker deploys.
- **Login + forgot-password rate-limiting** (§18.5). Two sliding-window buckets per login attempt: per-username (10/15 min) and per-client-IP (20/15 min). Forgot-password is 3/15 min per email + 10/15 min per IP, recorded eagerly because the endpoint intentionally returns success for nonexistent emails (anti-enumeration). Behind a reverse proxy, set `TRUSTED_PROXY_IPS` (comma-separated) to enable right-to-left X-Forwarded-For resolution; unset = the direct TCP peer is used (safe default for single-host deploys). Multi-hop nginx chains are handled correctly.
- **Password complexity policy** (§18.6). Create / change / reset flows now require uppercase + lowercase + digit + special character on top of the existing 8-character minimum. Existing stored hashes are not re-validated (nobody gets locked out on upgrade), only new passwords are checked. Max-length caps added across the board (password 256, email 254 per RFC 5321, username 150, forgot-password token 128) to bound the CPU cost of a hostile input before it reaches pbkdf2 / pyotp / email lookup.
- **Login page redesigned around the step machine** (§18.11). Password step → 2FA method picker (TOTP / email / backup code) → verify; OIDC provider buttons render below the password form when providers are configured; forgot-password opens in a sub-modal on the same page. All copy available in English and Ukrainian including every OIDC error code the server can emit.
- **Auth regression test suite + three latent fixes** (§18.13). Ported a focused subset of upstream's auth tests — 50 tests across `test_client_ip.py` (X-Forwarded-For trusted-proxy parsing), `test_mfa_helpers.py` (backup codes + TOTP QR), `test_security.py` (all 8 maintainer-review coverage gaps: encryption round-trip, JWT revocation, OIDC exchange single-use, email_verified guard, email-OTP max-attempts lockout, OIDC SSRF scheme rejection, login IP+username rate-limit buckets, 2fa_challenge cookie binding) and `test_mfa_api.py` (happy-path subset of the 2FA lifecycle). Greening the tests surfaced three latent bugs that have been fixed in the same commit: (1) `POST /auth/login` did not detect active TOTP / email-OTP and always issued a JWT — the 2FA step machine existed in `mfa.py` but `/login` never triggered it; now returns `pre_auth_token` + 2FA challenge cookie when a method is enabled. (2) `GET /auth/me` had a custom JWT handler that bypassed `get_current_user`, so logout did not invalidate it; now checks JTI revocation + `_is_token_fresh`. (3) `security_headers_middleware` was registered before `auth_middleware`, so the X-Content-Type-Options / X-Frame-Options / Referrer-Policy headers never landed on 401/503 early returns; reordered so it now wraps everything.

## [0.3.1.1] - 2026-04-17

### 3MF Download Recovery Service

When ``on_print_start`` can't pull the 3MF from the printer's SD (flaky FTP, path mismatch, slow card, TLS hiccup) a fallback archive is created with ``file_path=""``.  Four on-demand recovery triggers fill it in later — no periodic polling, so short prints aren't affected.

- **Triggers**:
  1. **BamDude startup sweep** — one-shot pass over every ``status='printing' AND file_path=''`` archive on server start (runs as `asyncio.create_task` so lifespan doesn't block the API).
  2. **Printer reconnect** — `PrinterManager.connect_printer` fires `retry_printer_archives(printer_id)` after a successful connection.
  3. **`on_print_complete` last-chance** — right after the swap-macro block, before SD cleanup runs, we try one more download.  The file is still on SD and the printer is no longer busy writing — highest-probability success window.
  4. **Manual** — new `POST /api/v1/archives/{id}/retry-download` endpoint.  Frontend exposes a "Retry 3MF download" menu item on archive cards, visible only when `file_path` is empty.
- **Per-archive lock with skip semantics** — concurrent calls (e.g. UI button click while startup sweep is running) immediately return `"in_progress"` with an info-level toast instead of duplicate FTP sessions.  Five distinct return statuses (`recovered`, `already_has_file`, `in_progress`, `failed`, `error`) mapped to clean UX messages in EN+UK.
- **`ArchiveService.attach_3mf_to_archive(archive_id, source_file, original_filename)`** — updates an existing row in place: copies the file to a fresh archive dir, parses 3MF via `ThreeMFParser`, extracts thumbnail, fills `content_hash`, `thumbnail_path`, `print_name`, all filament/plate metadata fields, clears `no_3mf_available` and any retry metadata.  Uses `original_filename` for dest naming so the file isn't left as `cover_1_*` or similar temp-prefixed.

### FTP Download Hardening

Root cause of recurring retry failures: `download_file_async` wraps the transfer in `asyncio.wait_for(timeout=60.0)`, which killed in-flight 28 MB downloads from slow SD cards even when the printer was still sending bytes.

- **`archive_download.try_download_3mf` rewritten** to use `download_file_try_paths_async` — one FTP session, multi-path probe, no outer asyncio timeout (same code path the cover endpoint uses reliably).  Socket-level timeout from the `ftp_timeout` DB setting still applies.
- **Stage 1 remote-path order**: root (`/{filename}`) pulled to the front of the list (was last).  BamDude-dispatched prints upload to root, so the fallback flow now hits the right path on its first try instead of 4 timeouts on `/cache`, `/model`, `/data`, `/data/Metadata` before finding it.
- **Stage 2 fuzzy-match** also converted to single-session `download_file_try_paths_async`.

### Cover Endpoint — No FTP

`GET /printers/{id}/cover` previously downloaded the full 3MF (up to 30 MB) from the printer every time the UI asked for a thumbnail PNG.  This raced with the archive download-retry service (both pulling the same file via separate FTP sessions) and wasted bandwidth on repeat renders.

- **Rewritten to serve from local archive only**: resolve the printing `PrintArchive` for this printer + subtask, serve `archive.thumbnail_path` directly if available, else open the already-archived 3MF from `archive_dir` and extract the needed plate's PNG.
- **Returns 404** if no archive with a local 3MF exists yet — the UI falls back to a placeholder.  Once the retry service fills the archive (via any trigger), the next cover request succeeds.
- **Removed** `download_file_try_paths_async` import, temp-file lifecycle, `_cover_cache` still used for PNG-byte caching but keyed on `(subtask, plate, view)`.

### `on_print_start` Download Refactor

- Extracted the inline probe + fuzzy-search block (~115 lines) from `on_print_start` into shared `services/archive_download.py::try_download_3mf`.  Both the initial download and the background retry service call the same helper — previously they had divergent behaviour.
- Removed the 2-tier retry wrapper around each remote_path; the new single-session approach is faster AND more reliable.

### Firmware Check — Cloudflare TLS Bypass

Bambu Lab moved behind a Cloudflare tier that blocks plain httpx via TLS fingerprinting (JA3): httpx's Python SSL handshake differs from real Chrome, so `GET /en/support/firmware-download/all` returned 403 regardless of `User-Agent`.  Our buildId parse silently failed, fell back to wiki (which has no download URL), and the UI reported "Firmware download not yet available" for P1 etc. even when the file was listed on the public page.
